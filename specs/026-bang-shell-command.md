# 功能规格说明：Bang Shell 命令

**特性分支**：`026-bang-shell-command`
**创建日期**：2026-03-25

## 用户场景与测试 *（必填）*

### 用户故事 1 - 基本命令执行（优先级：P1）

作为用户，我希望能够通过 `!` 前缀直接从聊天输入执行 shell 命令，以便我可以快速执行系统操作而无需离开 agent 界面。

**为什么是这个优先级**：这是功能的核心。它为用户提供了与系统交互的强大方式。

**独立测试**：可以通过在聊天输入中输入 `!echo "hello"` 等命令并验证命令被执行且输出 "hello" 显示在对话中来测试。

**验收场景**：

1. **假设**聊天输入为空，**当**用户输入 `!ls -la` 并按下 Enter 时，**则**命令 `ls -la` 在当前工作目录中执行，其输出显示在 `BangBlock` 中。
2. **假设**命令正在执行中，**当**用户输入另一个命令时，**则**系统应该防止并发执行并显示错误消息。

---

### 用户故事 2 - 输出显示与截断（优先级：P2）

作为用户，我希望 shell 命令的输出清晰显示，长输出默认被截断以避免聊天历史混乱，但如果需要仍然可以查看完整结果。

**为什么是这个优先级**：通过保持聊天历史可读来确保良好的用户体验，同时仍提供对完整命令结果的访问。

**独立测试**：可以通过执行具有多行输出的命令（如 `!seq 1 100`）并验证默认仅显示最后几行，展开块可以显示完整输出来测试。

**验收场景**：

1. **假设**一个有 20 行输出的命令，**当**输出显示在聊天中时，**则**默认仅显示最后 3 行。
2. **假设**一个截断的输出块，**当**用户展开块时（如通过点击或使用键盘快捷键），**则**显示完整的 20 行输出。

---

### 用户故事 3 - 命令中止（优先级：P2）

作为用户，我希望能够中止长时间运行的 shell 命令，如果我意识到这是一个错误或它花费太长时间。

**为什么是这个优先级**：这防止 agent 卡在等待可能永远不会完成或用户不再想运行的命令上。

**验收场景**：

1. **假设**长时间运行的命令如 `!sleep 60` 正在执行，**当**用户触发中止操作（如 Ctrl+C）时，**则**进程被杀死，`BangBlock` 反映命令已终止。

---

### 边界情况

- **命令不存在怎么办？** 系统应该捕获 shell 的错误并在 `BangBlock` 中显示，带有适当的状态（如红色）。
- **命令没有产生输出怎么办？** `BangBlock` 仍应显示命令及其退出状态，但输出区域应为空或隐藏。
- **命令被信号中断（如 SIGINT）怎么办？** 系统应该优雅地处理信号，更新退出码（如 SIGINT 为 130），并显示命令已被中断。
- **命令在不存在的目录中执行怎么办？** `BangManager` 应该处理错误并向用户报告。

## 需求 *（必填）*

### 功能需求

- **FR-001**：系统必须检测以 `!` 开头的聊天输入作为 shell 命令。
- **FR-002**：系统必须在当前工作目录中执行 `!` 后面的命令。
- **FR-003**：系统必须捕获执行命令的 stdout 和 stderr。
- **FR-004**：系统必须在对话历史中的专用 `BangBlock` 中显示命令及其输出。
- **FR-005**：系统必须使用视觉提示（如颜色）显示执行状态（运行中、成功、失败）。
- **FR-006**：系统必须默认将长输出截断为最多 3 行。
- **FR-007**：系统必须提供展开 `BangBlock` 以显示完整输出的方式。
- **FR-008**：系统必须防止多个 bang 命令并发执行。
- **FR-009**：系统必须提供中止正在运行的 bang 命令的机制。

### 关键实体 *（如果功能涉及数据则包含）*

- **BangBlock**：表示 shell 命令执行的消息块类型，包含命令字符串、输出、运行状态和退出码。
- **BangManager**：负责生成 shell 进程、管理其生命周期以及向消息历史传递更新的服务。
