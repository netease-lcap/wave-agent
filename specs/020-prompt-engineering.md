# 功能规格说明：提示工程框架

**特性分支**：`020-prompt-engineering`  
**创建日期**：2026-03-25  

## 用户场景与测试 *（必填）*

### 用户故事 1 - 集中式提示管理（优先级：P1）

作为开发者，我希望以集中和结构化的方式管理所有系统提示和工具描述，以便我可以轻松更新、版本化和优化它们而不修改核心逻辑。

**为什么是这个优先级**：这是该功能的核心价值，实现代理行为的更好可维护性和迭代。

**独立测试**：可以通过验证所有提示被移动到专用提示管理系统且代理仍然行为正确来测试。

**验收场景**：

1. **假设**有一组系统提示和工具描述，**当**我将它们移动到提示工程框架时，**则**代理应该在初始化期间仍然接收正确的提示
2. **假设**框架已就位，**当**我在框架中更新提示时，**则**代理应该在下一次会话中立即使用更新后的提示

---

### 用户故事 2 - 动态工具描述（优先级：P1）

作为开发者，我希望基于当前上下文（如可用子代理、技能或工作目录）提供动态工具描述，以便代理拥有工具选择的最相关信息。

**为什么是这个优先级**：这已经通过工具中的 `prompt()` 部分实现，但在框架内形式化将使其更健壮和易于管理。

**独立测试**：可以通过验证工具描述基于提供的上下文变化来测试。

**验收场景**：

1. **假设**带有动态提示的工具，**当**工具以不同上下文注册时，**则**工具在 OpenAI 函数调用配置中的描述应该反映每个上下文的正确提示

---

### 用户故事 3 - System-Reminder 消息注入（优先级：P1）

作为开发者，我希望将临时指令作为 `<system-reminder>` 包装的用户消息（而非系统提示更改）注入对话，以便系统提示在模式切换间保持不变，保持提示缓存并减少 token 成本。

**为什么是这个优先级**：在会话中途修改系统提示（如附加计划模式指令）会在每次模式更改时失效整个缓存的系统提示前缀。将指令作为带 `<system-reminder>` 标签的用户消息注入保持缓存同时仍然向模型传递上下文指令。此模式被 Claude Code 用于所有动态模式特定指导。

**独立测试**：进入计划模式，验证系统提示与上一轮相同且计划模式指令作为 `<system-reminder>` 用户消息出现在 API 请求中。

**验收场景**：

1. **假设**代理进入计划模式，**当**下一个 API 请求被组装时，**则**计划模式指令必须作为 `<system-reminder>` 包装的用户消息注入，而非附加到系统提示
2. **假设**代理退出计划模式，**当**下一个 API 请求被组装时，**则**"退出计划模式"的 `<system-reminder>` 用户消息必须被注入（仅一次）
3. **假设**代理在退出后重新进入计划模式，**当**计划文件已存在时，**则**重新进入的 `<system-reminder>` 用户消息必须被注入，指示模型读取现有计划并评估是继续还是重新开始
4. **假设**系统提示已被先前请求缓存，**当**模式切换发生时，**则**系统提示必须保持字节相同，启用系统消息前缀的缓存命中

---

### 用户故事 4 - 节流提醒注入（优先级：P2）

作为开发者，我希望计划模式提醒被节流，以便它们仅每 N 个人类轮次注入一次（不是每个工具轮次），以减少 token 浪费同时保持约束意识。

**为什么是这个优先级**：没有节流，计划模式提醒在每次 API 调用（包括单个人类轮次内的工具轮次）时注入，浪费 token。节流到每 5 个人类轮次减少成本，同时定期完整提醒防止模型忘记约束。

**独立测试**：在计划模式下工作 10+ 轮，验证提醒每 5 个人类轮次出现一次，在完整版本和稀疏版本之间交替。

**验收场景**：

1. **假设**计划模式活跃且提醒刚被注入，**当**不到 5 个人类轮次过去时，**则**不注入计划模式提醒
2. **假设**自上次提醒以来已过去 5 个人类轮次，**当**下一个 API 请求发出时，**则**注入计划模式提醒
3. **假设**每第 5 次提醒注入，**当**提醒被注入时，**则**它必须是完整的计划模式指令（完整的 5 阶段工作流）
4. **假设**非第 5 次提醒注入，**当**提醒被注入时，**则**它必须是引用先前完整指令的简短稀疏提醒

---

### 用户故事 5 - 模式切换的覆盖语言（优先级：P1）

作为开发者，我希望在模式切换后注入的 `<system-reminder>` 指令包含明确的覆盖语言（如"This supercedes any other instructions you have received"），以便模型理解新约束优先于对话中先前的工具调用历史。

**为什么是这个优先级**：当在对话中途从 default/acceptEdits 模式切换到计划模式时，消息历史包含最近的 Edit/Write 工具调用，可能误导模型继续编辑。覆盖语言加上提醒是模型看到的最近指令，确保模型遵守新模式约束。

**独立测试**：进行包含 Edit/Write 工具调用的对话，然后进入计划模式，验证计划模式提醒包含"supercedes"覆盖语言并出现在消息流中所有先前工具调用之后。

**验收场景**：

1. **假设**对话包含最近的 Edit/Write 工具调用，**当**用户进入计划模式时，**则**计划模式 `<system-reminder>` 必须作为 API 请求中的最后一条用户消息注入（在所有先前工具调用之后）
2. **假设**计划模式提醒被注入，**当**模型读取它时，**则**提醒必须包含"This supercedes any other instructions you have received"或等效覆盖语言
3. **假设**计划模式提醒在模式切换后被注入，**当**模型尝试对计划文件以外的任何文件使用 Edit 或 Write 时，**则**权限系统在运行时阻止该操作（纵深防御）

---

### 边界情况

- **缺失的提示**：当框架中缺少必需提示时会发生什么？（系统应该回退到默认提示或显示清晰的错误消息）
- **大型提示**：系统如何处理可能超过 token 限制的非常大提示？（框架应该提供提示压缩或截断工具）
- **冲突的提示**：系统如何处理来自不同提示来源的冲突指令？（框架应该定义明确的优先级顺序）
- **快速模式切换**：当用户在计划和非计划模式之间快速切换时，进入和退出的 `<system-reminder>` 消息不应同时注入。重新进入计划模式时应清除退出附加标志
- **压缩后重新进入**：当压缩移除所有对话历史包括先前的 `<system-reminder>` 消息时，系统必须重新注入完整的计划模式提醒（非稀疏），因为没有更早的完整指令存在

## 需求 *（必填）*

### 功能需求

- **FR-001**：系统必须为所有系统提示提供集中注册表
- **FR-002**：系统必须支持基于执行上下文的动态提示生成
- **FR-003**：系统必须允许工具提供自己的提示以覆盖或增强其描述
- **FR-004**：系统必须支持提示版本化和 A/B 测试
- **FR-005**：系统必须提供根据 token 限制验证提示的方式
- **FR-006**：系统必须支持带变量替换的提示模板
- **FR-007**：系统必须整合 Claude Code 的最佳实践（如操作安全性、协作者心态、简洁输出）
- **FR-008**：系统必须将模式特定指令（如计划模式、计划模式退出、计划模式重新进入）作为 `<system-reminder>` 包装的用户消息注入而非系统提示修改，以保持跨模式切换的缓存系统提示前缀
- **FR-009**：所有 `<system-reminder>` 注入的消息必须使用 `isMeta: true` 且不得在 UI 中渲染
- **FR-010**：系统必须在计划模式 `<system-reminder>` 消息中包含覆盖语言（"This supercedes any other instructions you have received"），以确保模型遵守新模式约束，尽管有先前的工具调用历史
- **FR-011**：系统必须将计划模式 `<system-reminder>` 注入节流到每 5 个人类轮次（非元数据、非工具结果的 user 消息），不是每个工具轮次。每第 5 次注入的提醒必须是完整指令；中间提醒必须是稀疏的
- **FR-012**：当计划模式活跃时，系统必须在压缩后重新注入完整的（非稀疏）计划模式 `<system-reminder>`，因为先前的提醒已被压缩移除
- **FR-013**：系统必须追踪 `hasExitedPlanMode` 和 `needsPlanModeExitAttachment` 状态标志，用于重新进入和退出通知 `<system-reminder>` 消息的一次性注入

### 关键实体 *（如果功能涉及数据则包含）*

- **提示注册表**：所有提示模板和配置的中央存储
- **提示模板**：可以用上下文数据填充的带占位符的字符串
- **执行上下文**：关于当前会话、可用工具和环境的数据，用于填充提示模板
- **工具提示**：工具插件提供的动态描述
