# 功能规格说明：Markdown 渲染系统

**特性分支**：`018-markdown-rendering-system`  
**创建日期**：2024-12-01  

## 用户场景与测试 *（必填）*

### 用户故事 1 - 富文本显示（优先级：P1）

作为用户，我希望看到代理响应以标题、粗体文本和列表格式化，以便信息在终端中易于阅读和浏览。

**为什么是这个优先级**：对代理响应的可读性至关重要，代理响应通常包含结构化信息。

**独立测试**：可以向 `Markdown` 组件提供包含标题、粗体、斜体和列表的 markdown 字符串，并验证它们以适当的 Ink 样式（粗体、颜色、缩进）渲染。

**验收场景**：
1. **假设**markdown 字符串包含 `# Heading`，**当**渲染时，**则**它以粗体青色出现并带 `#` 前缀
2. **假设**markdown 字符串包含 `**bold**` 和 `*italic*`，**当**渲染时，**则**它们分别以粗体和斜体样式出现
3. **假设**markdown 字符串包含无序或有序列表，**当**渲染时，**则**它以适当的缩进和项目符号/编号出现

---

### 用户故事 2 - 代码块渲染（优先级：P1）

作为开发者，我希望看到代码块与常规文本清晰分离，支持语言提示和适当的间距。

**为什么是这个优先级**：Wave Agent 频繁生成和显示代码，因此清晰的代码块渲染至关重要。

**独立测试**：可以向 `Markdown` 组件提供包含围栏代码块（如 ```typescript ... ```）的 markdown 字符串，并验证它以带灰色分隔符的内边距框渲染。

**验收场景**：
1. **假设**带语言提示的围栏代码块，**当**渲染时，**则**开头和结尾的反引号以灰色显示，内容带内边距
2. **假设**没有语言提示的代码块，**当**渲染时，**则**它以简单文本块显示

---

### 用户故事 3 - 响应式表格渲染（优先级：P2）

作为用户，我希望看到数据表格在终端中正确渲染，即使它们很宽，以便我可以查看结构化数据而不破坏布局。

**为什么是这个优先级**：表格是展示结构化数据的常见方式，而终端宽度是一个重要约束。

**独立测试**：可以提供包含不同内容长度的 markdown 表格并验证它以边框渲染且列按比例适应终端宽度。

**验收场景**：
1. **假设**markdown 表格，**当**渲染时，**则**它以单线边框和适当的列对齐出现
2. **假设**超出终端宽度的表格，**当**渲染时，**则**列宽按比例缩小以适应可用空间

---

## 需求 *（必填）*

### 功能需求

- **FR-001**：系统必须使用 `marked` 库将 Markdown 字符串解析为 token
- **FR-002**：系统必须渲染内联元素：text、strong（粗体）、em（斜体）、codespan（内联代码）、link（同时显示文本和 URL）、br（换行）和 del（删除线）
- **FR-003**：系统必须渲染块元素：标题（H1-H6）、段落、围栏代码块、列表（有序和无序）、引用块和水平线
- **FR-004**：系统必须渲染带边框和表头的表格
- **FR-005**：系统必须基于内容自动计算表格的列宽
- **FR-006**：如果总宽度超过终端宽度，系统必须按比例缩放表格列
- **FR-007**：系统必须通过反转义处理 markdown 中的 HTML 实体（如 `&lt;` 转为 `<`）
- **FR-008**：系统必须支持嵌套元素（如列表项中的粗体文本）

### 关键实体

- **Markdown 组件**：渲染 markdown 字符串的主入口点
- **BlockRenderer**：负责渲染块级 markdown token 的组件
- **InlineRenderer**：负责渲染内联 markdown token 的组件
- **TableRenderer**：专门用于渲染带响应式宽度计算的 markdown 表格的组件
