# 功能规格说明：消息渲染系统

**特性分支**：`025-message-rendering-system`
**创建日期**：2026-03-24

## 用户场景与测试 *（必填）*

### 用户故事 1 - 消息历史显示（优先级：P1）

作为用户，我希望看到与 agent 的对话历史，以便我可以回顾之前的交互。

**为什么是这个优先级**：对于基于聊天的 agent 的核心用户体验至关重要。

**独立测试**：向 `MessageList` 组件提供消息列表并验证它们按时间顺序渲染。

**验收场景**：
1. **假设**有消息列表，**当**渲染时，**则**它们按时间顺序显示。
2. **假设**有较长的消息历史，**当**渲染时，**则**仅显示最近的消息以维护性能和可读性。
3. **假设**有历史消息，**当**渲染时，**则**它们使用 Ink 的 `Static` 组件渲染以避免不必要的重新渲染。

---

### 用户故事 2 - 动态块渲染（优先级：P1）

作为用户，我希望看到活动工具执行或运行命令的实时更新，以便我知道 agent 正在工作。

**为什么是这个优先级**：在长时间运行的操作期间为用户提供即时反馈。

**独立测试**：将工具块的阶段从 `running` 更新为 `end` 并验证 UI 相应更新。

**验收场景**：
1. **假设**一个处于 `running` 阶段的工具块，**当**渲染时，**则**它显示动态展示（如加载指示器或进度指示器）。
2. **假设**一个正在运行的 `bang` 命令，**当**渲染时，**则**其输出实时更新。
3. **假设**任何其他块类型（如 text、error），**当**渲染时，**则**它被视为静态内容。

---

### 用户故事 3 - 多种块类型支持（优先级：P1）

作为用户，我希望不同类型的内容（文本、代码、错误、图片、工具调用）被适当地渲染，以便我可以轻松区分它们。

**为什么是这个优先级**：确保复杂 agent 响应的清晰度和可读性。

**独立测试**：渲染包含每种块类型之一的消息并验证它们都正确显示。

**验收场景**：
1. **假设**一个 `stage === "streaming"` 的文本块，**当**在折叠模式下渲染时，**则**它显示最后 30 个字符，灰色并截断（与工具和推理块流式行为一致）。
2. **假设**一个 `stage !== "streaming"`（如 `end` 或未定义）的文本块，**当**渲染时，**则**它使用 Markdown 渲染显示。
3. **假设**用户消息中的文本块，**当**渲染时，**则**它显示完整内容，无论 stage 如何。
4. **假设**一个错误块，**当**渲染时，**则**它以红色显示并带有 "Error:" 前缀。
5. **假设**一个工具块，**当**渲染时，**则**它使用专门的 `ToolDisplay` 组件。
6. **假设**一个图片块，**当**渲染时，**则**它显示图片的占位符或摘要。

---

## 边界情况

- **消息超过 10 条怎么办？** 系统应该只渲染最后 10 条消息，以确保终端保持响应且回滚不会变得过多。
- **`forceStatic` 为 true 或视图被展开怎么办？** 所有块，包括最后一条消息中的块，都应该渲染为静态内容，禁用任何动态更新或加载指示器。这确保视图在展开时保持"冻结"和高性能。
- **块类型未知怎么办？** 系统应该优雅地处理，通过忽略它或显示通用占位符，以防止整个 UI 崩溃。

## 需求 *（必填）*

### 功能需求

- **FR-001**：系统必须按时间顺序渲染 `Message` 对象列表。
- **FR-002**：系统必须将消息扁平化为单独的 `MessageBlock` 项进行渲染。
- **FR-003**：系统必须使用 Ink 的 `Static` 组件渲染历史（非动态）消息块。
- **FR-004**：系统必须识别"动态"块并在 `Static` 组件之外渲染它们。
- **FR-004.1**：如果 `forceStatic` 为 false 且 `isExpanded` 为 false 且块属于最后一条消息且消息包含至少一个活动块（`tool` 处于 `running`/`streaming`/`start` 阶段，`bang` 处于 `running` 阶段，或 `text` 处于 `streaming` 阶段）且块本身不是已完成的 `text` 或 `reasoning` 块（`stage === "end"`），则该块是动态的。
- **FR-004.2**：所有其他块必须渲染为静态内容，包括 `stage === "end"` 的 text/reasoning 块，即使同一消息中的其他块仍处于活动状态。
- **FR-004.3**：不在最后一条消息中的块必须始终为静态，无论其消息是否包含活动块。
- **FR-005**：系统必须在消息列表顶部支持显示版本和环境信息的"欢迎消息"。
- **FR-006**：系统必须将渲染的消息数量限制为折叠模式下最多 30 条，展开模式下最多 10 条。
- **FR-007**：系统必须提供测量动态块高度并通过回调报告的机制。
- **FR-008**：系统必须支持消息块的展开和折叠视图。

### 关键实体

- **MessageList**：管理消息历史和活动块渲染的主容器组件。
- **MessageBlockItem**：根据块类型将渲染分派到专门组件的组件。
- **BlockWithStatus**：用渲染元数据（如 `isDynamic`、`key`）增强 `MessageBlock` 的内部数据结构。
