# 功能规格说明：支持 AskUserQuestion 工具

**特性分支**：`037-ask-user-tool`
**创建日期**：2026-01-19

## 用户场景与测试 *（必填）*

### 用户故事 1 - 澄清模糊指令（优先级：P1）

作为 AI agent，当我收到用户的模糊指令时，我希望使用结构化的多选界面提出澄清问题，以便我可以在正确理解用户意图的情况下继续。

**为什么是这个优先级**：这是工具的核心价值。它防止 agent 做出错误假设并提高交互质量。

**独立测试**：可以通过给 agent 一个模糊任务（如"重构这个函数"）并验证它使用 `AskUserQuestion` 工具询问期望的重构模式而不是猜测来测试。

**验收场景**：

1. **假设** agent 正在处理任务，**当**遇到模糊性时，**则**它必须调用 `AskUserQuestion` 工具并提供相关选项。
2. **假设**用户看到问题，**当**用户选择选项时，**则** agent 必须收到答案并基于该选择继续任务。

---

### 用户故事 2 - 在实现方案之间选择（优先级：P2）

作为 AI agent，当有多种有效方式实现功能时，我希望向用户展示这些选项，以便他们可以决定哪种方案最适合其需求。

**为什么是这个优先级**：使用户能够在不必自己编写代码的情况下做出架构或设计决策。

**独立测试**：可以通过要求 agent "添加认证" 并验证它询问是使用 JWT、OAuth2 还是基于 Session 的认证来测试。

**验收场景**：

1. **假设** agent 处于规划或实施阶段，**当**识别出多种方案时，**则**它应该使用 `AskUserQuestion` 让用户选择。
2. **假设**选项中包含推荐项，**则** agent 应该在标签中标记为 "(Recommended)"。

---

### 用户故事 3 - 在计划模式中收集需求（优先级：P2）

作为计划模式中的 AI agent，我希望在最终确定计划之前向用户询问缺失的需求，确保计划准确并获得批准。

**为什么是这个优先级**：确保规划阶段是交互式的并产生高质量计划。

**独立测试**：可以通过带着模糊请求进入计划模式并验证 agent 在调用 `ExitPlanMode` 之前提出问题来测试。

**验收场景**：

1. **假设** agent 处于计划模式，**当**需求缺失时，**则**它必须使用 `AskUserQuestion` 来收集。
2. **假设** agent 准备最终确定计划，**则**它不得将 `AskUserQuestion` 用于计划批准（必须使用 `ExitPlanMode` 代替）。

---

### 边界情况

- **用户提供自定义输入**：当用户选择 "Other" 并提供不在选项中的文本时会发生什么？系统必须将其作为有效答案字符串处理。
- **多个问题**：系统如何处理一次最多 4 个问题？UI 必须清晰显示它们并收集所有答案。
- **多选答案**：多个选中选项如何返回？应该以逗号分隔的字符串或类似的结构化格式返回。
- **工具拒绝**：如果用户拒绝回答怎么办？agent 应该被通知并决定是使用假设继续还是再次询问。

## 需求 *（必填）*

### 功能需求

- **FR-001**：系统必须提供名为 `AskUserQuestion` 的内置工具。
- **FR-002**：工具必须接受 1 到 4 个问题的数组。
- **FR-003**：每个问题必须有清晰的问题文本和简短标题（最多 12 个字符）。
- **FR-004**：每个问题必须提供 2 到 4 个多选选项。
- **FR-005**：每个选项必须有标签和可选的描述。
- **FR-006**：工具必须支持每个问题的 `multiSelect` 标志以允许多个答案。
- **FR-007**：UI 必须为每个问题提供 "Other" 选项以允许自定义文本输入。
- **FR-008**：工具必须将用户的答案作为从问题文本到所选答案的映射返回。
- **FR-009**：在计划模式中，agent 必须被指示使用 `AskUserQuestion` 进行澄清，但不用于计划批准。
- **FR-010**：必须阻止 agent 通过纯文本提出问题，并鼓励使用此工具代替。
- **FR-011**：当 `permissionMode` 设置为 `bypassPermissions` 时，`AskUserQuestion` 不得可用。

### 关键实体 *（如果功能涉及数据则包含）*

- **Question**：表示对用户的单个询问。
    - `question`：问题的完整文本。
    - `header`：问题的简短标识符。
    - `options`：预定义选择列表。
    - `multiSelect`：是否允许多选。
- **Option**：问题中的单个选择。
    - `label`：显示给用户的文本。
    - `description`：选择的附加上下文。
- **Answer**：用户对问题的回答。
    - `questionText`：被回答的问题文本。
    - `selectedOption`：选中选项的标签或自定义文本。

## 假设

- 渲染这些问题的 UI 组件已存在或将作为 `code` 包的一部分实现。
- `AskUserQuestion` 工具将被视为需要用户交互的"受限"工具，类似于 `Bash` 或 `Write`。
- "Other" 选项由 UI 自动处理，agent 不需要在工具的输入模式中明确定义。
