# 功能规格说明：/goal 命令

**特性分支**：`056-goal-command`
**创建日期**：2026-06-07

## 用户场景与测试 *（必填）*

### 用户故事 1 - 设置自治目标（优先级：P1）

作为用户，我希望为会话设置完成条件，以便 agent 跨轮次自治工作直到条件满足，而无需每轮手动提示。

**为什么是这个优先级**：这是核心功能——"设置后忘记"的自治工作。

**独立测试**：运行 `/goal all tests in test/auth pass` 并验证 agent 继续自治工作直到目标达成或断路器触发。

**验收场景**：

1. **假设** agent 处于空闲状态，**当**我输入 `/goal all tests in test/auth pass` 时，**则** agent 设置目标，显示确认并开始自治工作。
2. **假设**目标处于活动状态，**当** agent 完成一轮时，**则**快速模型使用对话记录评估目标条件是否满足。
3. **假设**评估器确定目标未满足，**当** agent 完成评估时，**则** agent 继续下一轮，包含原因和目标条件的系统提醒。
4. **假设**评估器确定目标已满足，**当** agent 完成评估时，**则**目标被清除，添加成就消息，Stop hooks 正常运行。

---

### 用户故事 2 - 检查目标状态（优先级：P2）

作为用户，我希望检查当前目标状态，以便查看条件、经过时间、轮次计数和最后评估原因。

**为什么是这个优先级**：提供对自治循环的可观察性而不中断它。

**独立测试**：在有活动目标的情况下运行 `/goal` 并验证状态消息包含条件、经过时间、轮次计数和最后原因。

**验收场景**：

1. **假设**目标处于活动状态，**当**我输入 `/goal` 时，**则**系统显示目标条件、经过时间、轮次计数和最后评估原因。
2. **假设**没有活动目标，**当**我输入 `/goal` 时，**则**系统显示"No active goal"并附带设置目标的说明。

---

### 用户故事 3 - 清除目标（优先级：P2）

作为用户，我希望取消活动目标，以便 agent 停止自治循环并返回正常交互模式。

**为什么是这个优先级**：用户对自治循环的控制至关重要。

**独立测试**：运行 `/goal clear` 并验证目标被停用且 agent 返回正常模式。

**验收场景**：

1. **假设**目标处于活动状态，**当**我输入 `/goal clear`（或 `stop`、`off`、`reset`、`none`、`cancel`）时，**则**目标被清除且 agent 返回正常交互模式。
2. **假设**没有活动目标，**当**我输入 `/goal clear` 时，**则**系统显示"No active goal to clear"。

---

### 边界情况

- **计划模式中的目标**：如果我尝试在计划模式中设置目标，系统应以错误消息拒绝，说明目标不能在计划模式中运行。
- **断路器——最大轮次**：如果目标循环达到 50 轮，系统应强制清除目标并显示"maximum turns exceeded"消息，然后转到 Stop hooks。
- **断路器——最大持续时间**：如果目标已活动 30 分钟，系统应强制清除并显示"time limit exceeded"，然后转到 Stop hooks。
- **断路器——评估失败**：如果连续 3 次快速模型评估调用失败，系统应强制清除目标并显示错误消息。
- **目标替代 Stop hooks**：当目标活动时，Stop hooks 被跳过（目标评估器替代它们）。在目标达成或强制清除时，Stop hooks 在最后一轮正常运行。
- **子 agent 隔离**：子 agent 完全跳过目标评估——只有顶级 agent 评估目标。
- **通知优先**：每轮目标评估前先排空待处理的后台任务通知。
- **/clear 清除目标**：运行 `/clear` 也会清除任何活动目标。
- **条件长度限制**：超过 4000 字符的目标条件被拒绝。

## 需求 *（必填）*

### 功能需求

- **FR-001**：系统必须支持 `/goal <condition>` 设置自治目标。agent 立即开始工作。
- **FR-002**：系统必须支持 `/goal`（无参数）显示当前目标状态，包括条件、经过时间、轮次计数和最后评估原因。
- **FR-003**：系统必须支持 `/goal clear|stop|off|reset|none|cancel` 停用活动目标。
- **FR-004**：每次 AI 轮次后（在 recursionDepth=0），如果有活动目标且不在子 agent 中，系统必须使用快速模型评估目标。
- **FR-005**：目标评估必须绕过 1 QPS 速率限制器（直接非流式调用，不使用 `acquireSlot`）。
- **FR-006**：目标评估必须通过 `convertMessagesForAPI()` 传递对话消息（与 compact 相同），剥离图像以减少 token 使用。
- **FR-007**：系统必须使用 `operation_type: "goal_evaluation"` 单独跟踪评估 token，不与 agent token 混合。
- **FR-008**：系统必须实现断路器：最多 50 轮、最多 30 分钟持续时间、最多 3 次连续评估失败。
- **FR-009**：当目标活动时，目标评估必须替代 Stop hooks。在目标达成或强制清除时，Stop hooks 正常运行。
- **FR-010**：系统必须拒绝在计划模式中设置目标。
- **FR-011**：系统必须拒绝超过 4000 字符的目标条件。
- **FR-012**：`/clear` 必须同时清除任何活动目标。
- **FR-013**：目标状态仅保存在内存中。不跨进程重启持久化。
- **FR-014**：当目标活动时，UI 状态行必须以青色显示 `◎ /goal active (<elapsed>)`。
- **FR-015**：目标评估必须在评估前排空待处理通知，以便评估器看到新鲜结果。
- **FR-016**：在目标轮次之间，加载状态必须保持活动以防止 UI 闪烁。
- **FR-017**：UI 必须在目标评估期间显示加载指示器（`✻ Evaluating goal...`）。

### 关键实体

- **GoalState**：活动目标的内存状态。
    - **condition**：`string` — 用户的目标条件（最多 4000 字符）。
    - **startedAt**：`number` — 目标设置时的时间戳。
    - **turnCount**：`number` — 每个评估周期递增。
    - **tokenBaseline**：`number` — 目标开始时的总 token 数。
    - **lastReason**：`string | undefined` — 评估器的最后原因。
    - **consecutiveEvalFailures**：`number` — 断路器计数器。
