# 功能规格说明：更新命令

**特性分支**：`029-update-command`
**创建日期**：2026-03-31

## 用户场景与测试 *（必填）*

### 用户故事 1 - 检查更新（优先级：P1）

作为用户，我希望能够检查 WAVE Code 是否有新版本可用，以便我可以保持最新的功能和错误修复。

**为什么是这个优先级**：这是功能的核心。它允许用户知道何时可用更新。

**独立测试**：可以通过运行 `wave update` 并验证它正确识别当前版本和 npm 注册表上的最新版本来测试。

**验收场景**：

1. **假设**当前版本是 `0.11.6`，npm 上的最新版本是 `0.11.6`，**当**用户运行 `wave update` 时，**则**系统通知用户 WAVE Code 已是最新。
2. **假设**当前版本是 `0.11.5`，npm 上的最新版本是 `0.11.6`，**当**用户运行 `wave update` 时，**则**系统通知用户有新版本可用。

---

### 用户故事 2 - 执行更新（优先级：P1）

作为用户，我希望工具使用适当的包管理器自动更新到最新版本，这样我就不必手动运行安装命令。

**为什么是这个优先级**：这为用户提供无缝的更新体验。

**独立测试**：可以在有更新可用时运行 `wave update` 并验证它执行正确的更新命令（如 `npm install -g wave-code@latest`）来测试。

**验收场景**：

1. **假设**有新版本可用且工具通过 `npm` 安装，**当**用户运行 `wave update` 时，**则**系统执行 `npm install -g wave-code@latest`。
2. **假设**有新版本可用且工具通过 `pnpm` 安装，**当**用户运行 `wave update` 时，**则**系统执行 `pnpm add -g wave-code@latest`。

---

### 边界情况

- **网络不可用怎么办？** 系统应该优雅地处理错误并通知用户无法检查更新。
- **更新命令失败（如权限问题）怎么办？** 系统应该通知用户失败原因并提供手动运行的命令（可能需要 `sudo`）。
- **工具未全局安装怎么办？** 系统仍应尝试检测包管理器并执行更新，或在无法做到时提供说明。

## 需求 *（必填）*

### 功能需求

- **FR-001**：系统必须提供 `update` 命令。
- **FR-002**：系统必须从 npm 注册表（`https://registry.npmjs.org/wave-code/latest`）获取 `wave-code` 的最新版本。
- **FR-003**：系统必须使用 semver 逻辑比较当前版本与最新版本。
- **FR-004**：系统必须检测用于安装的包管理器（`npm`、`pnpm` 或 `yarn`）。
- **FR-005**：系统必须为检测到的包管理器执行适当的更新命令。
- **FR-006**：系统必须在更新过程中通过继承 stdio 提供实时反馈。
- **FR-007**：系统必须优雅地处理版本检查和更新执行过程中的错误。

### 关键实体 *（如果功能涉及数据则包含）*

- **Version**：表示工具的 semver 版本。
- **Package Manager**：用于管理 `wave-code` 安装的工具（npm、pnpm 或 yarn）。
