# 功能规格说明：Plan 子 Agent 支持

**特性分支**：`044-plan-subagent`
**创建日期**：2026-02-12

## 用户场景与测试 *（必填）*

### 用户故事 1 - 使用内置 Plan 子 agent 的计划模式（优先级：P1）

当用户进入计划模式时，他们可以生成内置的 Plan 子 agent 来探索代码库并设计实现方案。Plan 子 agent 充当软件架构师，基于彻底的代码探索提供详细的实现计划。

**为什么是这个优先级**：核心功能，通过提供专门设计实现策略而不做任何代码更改的 agent 来启用计划模式工作流。

**独立测试**：可以通过进入计划模式、生成带有需求的 Plan 子 agent，并验证子 agent 探索代码库并返回详细实现计划而不修改任何文件来进行完整测试。

**验收场景**：

1. **假设**用户处于计划模式，**当**用户生成带有实现需求的 Plan 子 agent 时，**则**子 agent 探索相关代码并返回包含关键文件识别的详细计划
2. **假设** Plan 子 agent 正在探索代码库，**当**子 agent 尝试编辑或创建文件时，**则**操作被阻止并返回错误
3. **假设**生成了多个具有不同视角的 Plan 子 agent，**当**它们完成工作时，**则**每个都基于其分配的视角（例如，简洁性 vs 性能）返回计划

---

### 用户故事 2 - 只读工具限制（优先级：P1）

Plan 子 agent 严格限制为只读操作，确保它们可以探索和分析代码但不能在计划阶段进行任何修改。

**为什么是这个优先级**：对于维护计划模式的完整性至关重要，防止在设计阶段意外更改代码。

**独立测试**：可以通过尝试各种文件修改操作（Write、Edit、有副作用的 Bash 命令）并验证全部被阻止来进行测试。

**验收场景**：

1. **假设** Plan 子 agent 正在运行，**当**子 agent 尝试使用 Write 工具时，**则**工具不可用且操作失败
2. **假设** Plan 子 agent 正在运行，**当**子 agent 尝试使用 Edit 工具时，**则**工具不可用且操作失败
3. **假设** Plan 子 agent 正在运行，**当**子 agent 尝试 bash 命令如 mkdir、touch 或 rm 时，**则**系统提供清晰的指导，说明只允许只读 bash 操作

---

### 用户故事 3 - 多规划视角（优先级：P2）

用户可以生成多个具有不同视角的 Plan 子 agent，为复杂任务探索各种实现方案。

**为什么是这个优先级**：通过考虑多种架构方案和权衡来实现对复杂问题的彻底分析。

**独立测试**：可以通过生成多个具有不同视角提示的 Plan 子 agent，并验证每个都基于其分配的重点生成不同的计划来进行测试。

**验收场景**：

1. **假设**有一个复杂的重构任务，**当**用户生成具有"简洁性"和"性能"视角的 Plan 子 agent 时，**则**每个子 agent 生成与其分配视角一致的计划
2. **假设**多个 Plan 子 agent 正在运行，**当**它们探索同一代码库时，**则**它们的探索互不干扰

---

### 用户故事 4 - 关键文件识别（优先级：P2）

Plan 子 agent 识别并列出实现其提议计划最关键的 3-5 个文件，帮助用户理解将在何处进行更改。

**为什么是这个优先级**：为实现阶段提供清晰指导，并帮助用户评估提议更改的范围和影响。

**独立测试**：可以通过验证 Plan 子 agent 输出包含"Critical Files for Implementation"部分及文件路径和简要说明来进行测试。

**验收场景**：

1. **假设** Plan 子 agent 完成探索，**当**它生成计划时，**则**输出包含 3-5 个关键文件及其路径和原因
2. **假设**关键文件已被识别，**当**用户审查计划时，**则**文件路径准确且原因解释了其重要性

---

### 边界情况

- 当 Plan 子 agent 遇到因权限无法读取的文件时会发生什么？
- Plan 子 agent 如何处理非常大的代码库（探索可能耗时）？
- 当 Plan 子 agent 引用的文件在计划期间被移动或删除时会发生什么？
- 系统如何处理多个 Plan 子 agent 同时尝试探索相同文件？
- 当 Plan 子 agent 系统提示太大超出 token 限制时会发生什么？

## 需求 *（必填）*

### 功能需求

- **FR-001**：系统必须提供内置的"Plan"子 agent，无需用户配置即可使用
- **FR-002**：Plan 子 agent 必须限制为只读工具（Glob、Grep、Read 和只读 Bash 命令）
- **FR-003**：Plan 子 agent 不得有文件修改工具的访问权限（Write、Edit、NotebookEdit）
- **FR-004**：Plan 子 agent 必须包含系统提示，明确说明只读限制和规划职责
- **FR-005**：系统必须允许并行生成多个具有不同视角的 Plan 子 agent
- **FR-006**：Plan 子 agent 必须生成包含"Critical Files for Implementation"部分的输出
- **FR-007**：Plan 子 agent 必须按照 plan.tmp.js 中描述的方式与计划模式工作流集成
- **FR-008**：当 Plan 子 agent 尝试禁止的操作时，系统必须提供清晰的错误消息
- **FR-009**：Plan 子 agent 必须支持使用 Glob、Grep 和 Read 工具不受限制地探索代码库
- **FR-010**：Plan 子 agent 必须能够执行只读 Bash 命令（ls、git status、git log、git diff、find、cat、head、tail）
- **FR-011**：系统必须在 Plan 子 agent 提示中包含强调只读模式的关键提醒
- **FR-012**：Plan 子 agent 必须默认使用"inherit"模型以匹配父 agent 的模型
- **FR-013**：系统必须在 Task 工具描述中列出 Plan 子 agent 并附带适当的"whenToUse"指导
- **FR-014**：Plan 子 agent 必须可被具有相同名称的用户配置子 agent 覆盖
- **FR-015**：系统必须验证 Plan 子 agent 在运行时仅接收只读工具访问
- **FR-016**：当 Plan 子 agent 收到计划模式提醒时，提醒不得指示子 agent 写入或编辑计划文件。相反，它必须告诉子 agent 以文本输出形式返回发现，由父 agent 写入计划文件。提醒可以包含计划文件路径以便在文件已存在时读取上下文。

## 成功标准

Plan 子 agent 功能被认为成功实现，当：

1. ✓ Plan 子 agent 出现在可用子 agent 列表中
2. ✓ 用户可以从计划模式或通过 Task 工具生成 Plan 子 agent
3. ✓ Plan 子 agent 可以使用只读工具探索代码库
4. ✓ 所有文件修改尝试都被阻止并显示清晰错误
5. ✓ Plan 子 agent 生成包含关键文件部分的计划
6. ✓ 多个 Plan 子 agent 可以并行运行而不冲突
7. ✓ 单元测试和集成测试通过且具有足够的覆盖率
8. ✓ 文档清楚说明了 Plan 子 agent 的能力和限制
